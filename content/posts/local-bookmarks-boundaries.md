资源页需要一个轻量收藏夹，又不希望游客注册账号。BlogLiaz 选择把资源 ID 保存在浏览器的 `localStorage` 中，页面刷新后还能读回来，服务器不需要保存访客数据。

这个方案适合公开链接的临时整理，但它不是云端收藏服务。

## 只保存稳定 ID

收藏数据没有重复保存标题、网址和说明，只记录资源 ID：

```json
["mdn-web", "robots-rfc", "openai-platform"]
```

资源内容仍由网站代码提供。以后修改标题或说明，收藏项会自动显示新内容；删除某条资源后，旧 ID 也不会制造一份过期副本。ID 一旦公开使用，就不应随意改名，否则游客已有的收藏会失去对应项。

## 首次渲染要等浏览器数据

服务端生成 HTML 时没有 `window.localStorage`。如果首屏直接假定收藏数量为零，浏览器接管页面后又立刻改成真实数量，文字可能闪动，还可能产生渲染不一致。

资源页用 `savedReady` 区分“还没有读取”和“已经读取但确实为空”。读取完成前显示占位符，收藏夹被打开时显示读取状态。这个小状态能把“空”和“未知”分开。

## 读取和写入都可能失败

存储内容可能被手动修改，也可能不是合法 JSON。隐私模式、浏览器策略或空间限制也可能让写入失败。读取时需要 `try...catch`，解析后还要确认结果是字符串数组。

写入失败不应阻止游客继续打开资源。BlogLiaz 仍会更新当前页面里的收藏状态，但不会承诺刷新后一定保留。若收藏承担订单、进度或付费权益，静默降级就不合适了。

## 当前设备意味着明确的边界

本地收藏不会自动同步到另一台电脑，也不会跟随 GitHub 账号。清除站点数据、换浏览器或使用无痕窗口后，收藏可能消失。同一台设备上的其他浏览器配置文件也各自独立。

因此页面直接写明“收藏只保存在当前设备，不需要登录”。这句话不是附加说明，它决定游客是否会把收藏夹当成长期可靠的数据仓库。

## 什么时候再增加导入导出

资源多起来后，可以先增加“复制收藏清单”或“导出 JSON”，仍然不需要账号。只有出现跨设备同步、多人共享或长期备份需求时，登录和数据库才带来实际收益。

对个人博客来说，本地收藏的维护成本很低，也不会记录游客浏览轨迹。它提供的是当前设备上的便利，不应被包装成永久保存。
