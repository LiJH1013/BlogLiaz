采集器拿到一个非 200 响应时，不能把它们都归成“请求失败”。重定向、限流、服务端故障和资源不存在，需要不同处理。HTTP 已经提供了大部分判断线索，采集逻辑应该保留这些语义。

## 状态码决定下一步动作

`301` 和 `308` 表示永久重定向，长期任务可以更新规范地址；`302` 和 `307` 更适合保留原地址，下次仍从原入口请求。`404` 通常不该连续重试，`401` 和 `403` 也不是增加次数就能解决的问题。

`429` 表示请求过多，响应可能带有 `Retry-After`。采集器应优先尊重服务端给出的等待时间，没有该字段时再使用带抖动的退避。`500`、`502`、`503` 和 `504` 可能是临时故障，但仍要设置次数和总时长上限。

## 重试还要看请求方法

GET 被定义为安全方法，重复读取通常不会要求服务器改变资源状态。POST 可能触发写入、生成任务或计费，网络断开后自动重试可能造成重复副作用。

只读采集器应尽量使用 GET 或 HEAD。必须调用带副作用的接口时，需要目标服务明确支持幂等键或重复请求，再考虑自动重试。客户端无法仅凭“上次没有收到响应”判断服务端是否执行成功。

## 条件请求省掉重复传输

响应包含 `ETag` 时，后续请求可以发送 `If-None-Match`；包含 `Last-Modified` 时，可以发送 `If-Modified-Since`。资源没有变化，服务器可返回 `304 Not Modified`，采集器继续使用本地缓存。

缓存内容需要连同规范 URL、抓取时间、验证标识和响应状态一起保存。只缓存正文，不保存它为何仍然有效，后续很难判断应该复用还是重新请求。

## 超时要拆开设置

连接目标服务器和等待响应正文是两段不同的等待。连接超时可以较短，读取大文件则可能需要更长时间。只设置一个很大的总超时，会让连接故障占住并发；只设置很短的统一超时，又会误伤正常的慢响应。

流式下载还要限制最大体积，并在读取前检查 `Content-Type`。预期 HTML 却收到压缩包或视频时，继续读完整响应只会浪费带宽和存储。

## HTTP 之外还有站点规则

HTTP 允许访问，不代表采集任务就适合高频运行。robots.txt、站点条款、认证边界和个人信息处理仍要单独检查。并发与间隔应按站点设置，缓存和去重可以减少重复访问。

进一步核对协议语义时，可以直接看 [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html)；robots.txt 的标准化规则在 [RFC 9309](https://www.rfc-editor.org/rfc/rfc9309) 中。
