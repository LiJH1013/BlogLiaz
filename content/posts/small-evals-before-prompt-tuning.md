提示词改完后随手试两三个问题，很容易只看到刚修好的例子。另一类输入可能已经退化，只是没有再次被打开。小型评测集能把常用场景和已知失败固定下来，每次改动都用同一批样本比较。

## 样本来自真实任务边界

第一版不需要追求很大，可以先收 15 到 30 条有代表性的输入。数量只是起点，重点是覆盖正常情况、边界情况和曾经失败的情况。

以文章信息提取为例，样本可以包含标题明确的正文、缺少日期的正文、带多个候选标题的页面，以及完全无关的内容。每条样本要保留预期字段或可接受范围，不能只存输入。

```json
{
  "input": "页面正文……",
  "expected": {
    "category": "前端",
    "has_title": true,
    "max_tags": 5
  }
}
```

## 判定要尽量接近程序需求

结构化输出可以先检查 JSON 是否可解析、字段是否齐全、类型是否正确，再比较业务约束。摘要和改写这类开放结果不适合只做字符串相等，可以拆成事实是否保留、长度是否合规、是否出现禁用内容。

自动评分器适合处理明确规则。需要模型评分时，判定说明要给出正反例，并定期抽查评分结果。模型评分不能替代所有人工阅读，尤其是样本很少或内容风险较高时。

## 每次改动保留同一份对照

评测记录至少包括提示词版本、模型、参数、样本结果和失败原因。只记录总分，会看不到错误从哪里移动。一个版本从 24 条通过变成 26 条通过，也可能同时修好 4 条、弄坏 2 条。

失败项可以按结构错误、事实错误、遗漏和风格偏差分类。分类稳定后，才容易判断应该改提示词、改 schema、补输入上下文，还是在程序侧增加校验。

## 评测集也会过期

真实任务变化后，旧样本可能不再代表当前流量。新失败需要去重后补进评测集，已经废弃的业务规则也应移除。否则分数继续上升，实际体验却没有跟着变化。

涉及隐私的真实输入要先脱敏。无法安全保留的内容，可以重写成结构相同的合成样本，同时标记来源，避免以后误以为它是真实分布。

## 小型评测适合解决什么

它适合比较提示词版本、模型切换和结构校验规则，不足以证明系统在所有输入上都可靠。高风险操作还需要权限边界、人工确认和线上监控。

OpenAI 的 [评测最佳实践](https://developers.openai.com/api/docs/guides/evaluation-best-practices) 提供了评测设计的进一步说明；使用开源工具组织指标和数据集时，可以参考 [Hugging Face Evaluate](https://huggingface.co/docs/evaluate/a_quick_tour)。
